# NT36830 factory DTS vs mainline driver comparison

This note records the source-of-truth comparison used for
`panel-novatek-nt36830.c`.

## Sources

- Factory display data:
  - `android-fdt/dtbo-entry-05.dts`
  - node: `qcom,mdss_dsi_nt36830_wqhd_dualdsi_extclk_cmd`
  - first full node occurrence: line 9495 in the current extracted DTS
- Generated reference:
  - `panel-driver/generated-reference/nt36830_wqhd_dualdsi_extclk_cmd/`
  - generated by `scripts/generate-panel-driver.sh` from the pinned
    `linux-mdss-dsi-panel-driver-generator`
- Mainline candidate driver:
  - `panel-driver/panel-novatek-nt36830.c`
- Qualcomm downstream parser code found in the local factory/source cache:
  - `/home/dinochang/razorphone2linux/fw-extract-tmp/msm-4.9/drivers/video/fbdev/msm/mdss_dsi_panel.c`
  - `/home/dinochang/razorphone2linux/fw-extract-tmp/msm-4.9/drivers/gpu/drm/msm/dsi-staging/dsi_panel.c`

No Razer-specific downstream `nt36830.c` was found in the local source copies.
The factory Android path appears to be the normal Qualcomm downstream model:
generic MDSS/DSI display code parses panel timing and command arrays from DTS.
For this panel, the factory DTS command arrays are therefore treated as the
highest-priority source, while generated C is used as a decoder/reference.

## Qualcomm downstream C parser checks

The local downstream source confirms how the factory DTS is consumed:

| Downstream source | What was checked | Result for this port |
|---|---|---|
| `mdss_dsi_parse_dcs_cmds()` in `mdss_dsi_panel.c` | Reads `qcom,mdss-dsi-on-command` / `off-command`, converts `dsi_ctrl_hdr.dlen` with `ntohs()`, then stores packet payload pointers | Our raw packet decoder must treat payload length as big-endian; factory packet 21 is `f6 71` |
| `dsi_panel_parse_cmd_sets()` in DRM staging `dsi_panel.c` | Parses the same command packet arrays and command state properties | Confirms generator output is only a convenience decoder, not a higher authority than raw DTS |
| `mdss_dsi_panel_parse_dt()` / related helpers | Reads reset sequence, on/off command sets, timing, clock, and backlight control from DTS | No separate Razer C table is expected |
| `dsi_panel_parse_dsc_params()` / `mdss_dsi_parse_dsc_*` | Reads DSC slice size, bpc, bpp, block prediction, and encoder count from DTS | Driver DSC values should track the factory extclk command node |
| backlight parser | `bl_ctrl_wled` maps to WLED in downstream | Mainline backlight remains cautious; do not auto-raise brightness in normal boot |

## Command-sequence corrections from raw factory DTS

The raw `qcom,mdss-dsi-on-command` packet stream in the factory DTS decodes to
113 packets. The important packets checked against the driver are:

| Packet | Factory raw payload | Driver payload | Action |
|---:|---|---|---|
| 21 | `f6 71` | changed from `f6 70` to `f6 71` | Fixed |
| 107 | `51 0f ff` | `51 0f ff` | Keep driver value; generator presents this ambiguously as a helper argument |
| 111 | `11`, wait 120 ms | sent to DSI0 then DSI1 | OK |
| 112 | `29` | resets context to DSI0, then sends DSI0 and DSI1 | Verified |

The generated reference is useful, but it models only one `mipi_dsi_device`.
The production driver must keep the extra dual-DSI coordination code.

## Current known gaps

- The driver still uses a conservative 60 Hz DRM mode. Factory extclk command
  mode is a 120 Hz panel mode, but 120 Hz should be enabled only after the MSM
  DSI clock path is validated.
- DTS keeps MDSS/DSI disabled for the normal boot path (bootloader
  framebuffer / simpledrm). Enabling this driver should remain an explicit
  panel-bring-up experiment (`RAZER_DISPLAY_NATIVE_PANEL=1`).
- Backlight/WLED limits are still not finalized. Do not enable automatic high
  brightness in normal boot until Razer current/string limits are confirmed.

## 2026-07-06 re-verification (7.1 port)

Re-did the byte-for-byte on/off-command comparison against the raw msm-4.9
tree (`fw-extract-tmp/msm-4.9/arch/arm64/boot/dts/fih/RC2_common/
dsi-panel-nt36830-wqhd-dualmipi-extclk-cmd.dtsi`, `timing@0`, the 1440p node —
not just the flattened android-fdt DTBO dump) instead of relying on the
earlier decode, since the driver file that made it into the 7.1 branch came
from a different source path (codex/linux-7.1-display) than the one this
doc was written against.

- **Packet 21 (`0xf6`) had regressed back to `0x70`** in the 7.1-branch
  driver, i.e. the "Fixed" state recorded in the table above did not survive
  whatever produced the 7.1 driver file. Re-confirmed `0x71` is correct two
  ways: (a) manual decode of the raw factory byte stream, (b) the
  independently-generated `generated-reference/.../panel-nt36830-wqhd-
  dualdsi-extclk.c` (which decodes straight from the same DTS) also emits
  `0xf6, 0x71`. Fixed again; **if this regresses a third time, check whatever
  script/patch overwrites `panel-driver/panel-novatek-nt36830.c` during a
  branch adoption** — this file must always be hand-verified after adopting
  it from elsewhere, not assumed correct.
- Full packet-by-packet re-check: aside from the `0xf6` byte, all 110+
  packets in on-command and all 5 in off-command match the driver exactly
  (gamma LUTs, DSC page-10 registers, GIP timing, brightness/CABC/backlight
  registers, sleep/display-on sequencing).
- **Confirmed the no-PPS design against an explicit factory flag**: line 55
  of the factory dtsi sets `razer,mdss-dsi-disable-sending-pps;`. The
  generated reference driver does NOT know about this Razer-specific
  property — its `prepare()` unconditionally calls
  `mipi_dsi_picture_parameter_set()` + `mipi_dsi_compression_mode(true)`
  (standard MIPI DSC negotiation). Sending those on this panel would be
  wrong per the factory's own DTS. This is the clearest example so far of
  "generator output is a decoder, not an authority" from this repo's own
  rule of thumb — the generator gets this one detail wrong because it has
  no concept of the vendor override flag.
