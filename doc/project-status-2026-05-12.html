🔴 WiFi Debug 歷程

失敗 #1 缺少 tqftpserv

建置並安裝 ARM64 tqftpserv，建立 /readonly/firmware/image/wlanmdsp.mbn。
結果: tqftpserv 啟動成功，但從未收到 TFTP 請求，因為 modem root_pd 連 wlan_pd 還沒到就 crash 了。

失敗 #2–4 pd-mapper / wlan_pd / 停用 pd-mapper

各種 pd-mapper 排列組合均無效。crash 固定在 160~170ms，與 pd-mapper 無關。

失敗 #5 Module ABI mismatch NEW

裝置上的 /tmp/*.ko 與當前 kernel ABI 不符，導致所有 remoteproc 模組載入失敗，remoteproc0/3 根本不存在。
處理: WSL 重編所有模組，以 scp 單檔推送，depmod 重建 index。
結果: 模組全部載入成功，remoteproc3（modem）現在可以啟動。

進行中 #6 modem_fsg_oem_1/2 被拒 NEW

rmtfs log 確認：modem 啟動時請求 /boot/modem_fsg_oem_1 和 /boot/modem_fsg_oem_2，rmtfs 回 -1（分區未提供）。
但 關鍵發現：modem 在 fsg_oem 被拒後仍繼續初始化，成功完成 EFS 讀取（iovec read、bye from 0），crash 發生在 EFS 完成之後。
結論: fsg_oem 拒絕不是 crash 的直接原因，但 OEM NV item 缺失可能導致後續初始化失敗。

🔍 目前已確認的事實

  • Modem (remoteproc3 = 4080000.remoteproc) MBA + mpss 載入成功
  • Modem 在 QRTR 上成功註冊 node 0，service 66 (servreg) + 43 (subsystem control)
  • rmtfs EFS 讀取序列：modem_fs1 ✅ → modem_fs2 ✅ → modem_fsg ✅ → modem_fsc ✅ → modem_fsg_oem_1 ❌ → modem_fsg_oem_2 ❌，iovec 讀取完成後 crash
  • Crash 發生在 EFS session 結束（bye from 0）之後，不在 EFS 期間
  • Service 69 (WLFW/ath10k) 從未出現，wlan_pd 還沒機會載入
  • SLPI (remoteproc2) 缺少 slpi.mbn，無法啟動，可能是 modem crash 的原因之一
  • Coredump 無法取得（MBA booted without debug policy）
  • GPU: platform device 存在（qcom,adreno-630.2），a630_gmu.bin 存在，a630_sqe.fw 已傳入，CONFIG_DRM_MSM 已加入 config 但尚未刷入新 kernel

📦 slpi.mbn 提取指令

# WSL：確認 Android 分區有哪些
ls /dev/disk/by-partlabel/ | grep -iE "slpi|dsp|adsp"

# 嘗試直接從分區讀取（如果有 slpi_a/b 分區）
sudo dd if=/dev/disk/by-partlabel/slpi_a bs=4M | \
  strings | head -5   # 確認是 ELF/MBN

# 或從已掛載的 Android vendor/system 找
find /android 2>/dev/null -name "slpi.mbn" 2>/dev/null
find /vendor  2>/dev/null -name "slpi.mbn" 2>/dev/null

# 若找到，複製到韌體目錄
sudo cp /path/to/slpi.mbn /lib/firmware/qcom/sdm845/Razer/aura/slpi.mbn

📦 modem_fsg_oem 分區確認

# 確認裝置上有沒有對應的分區
ls /dev/disk/by-partlabel/ | grep -iE "oem|fsg"

# 若有 fsg_oem_1 之類的分區，加入 rmtfs
sudo rmtfs -v \
  -P /dev/disk/by-partlabel/modemst1 \
  -P /dev/disk/by-partlabel/modemst2 \
  -P /dev/disk/by-partlabel/fsg \
  -P /dev/disk/by-partlabel/<fsg_oem_1分區名> \
  -P /dev/disk/by-partlabel/<fsg_oem_2分區名> \
  > /tmp/rmtfs.log 2>&1 &

🚀 服務手動啟動順序（目前版本）

# 1. 載入模組（重開機後需重跑）
sudo modprobe qrtr qrtr-smd mdt_loader qcom_glink_smem
sudo modprobe qcom_pil_info qcom_common qcom_sysmon qcom_q6v5
sudo modprobe reset-qcom-pdc qcom_q6v5_pas qcom_q6v5_mss
sudo modprobe pdr_interface qcom_pdr_msg

# 2. pd-mapper（通常已 systemd 自啟）
sudo systemctl start pd-mapper

# 3. rmtfs（帶 fsg）
sudo rmtfs -v \
  -P /dev/disk/by-partlabel/modemst1 \
  -P /dev/disk/by-partlabel/modemst2 \
  -P /dev/disk/by-partlabel/fsg \
  > /tmp/rmtfs.log 2>&1 &
sleep 1

# 4. （有 slpi.mbn 後）啟動 SLPI
# echo start | sudo tee /sys/class/remoteproc/remoteproc2/state; sleep 2

# 5. 啟動 modem
echo start | sudo tee /sys/class/remoteproc/remoteproc3/state

# 確認
qrtr-lookup
cat /sys/class/remoteproc/remoteproc3/state
